# IntraAI RAG 系统综合评估报告

**评估日期**: 2026-06-16  
**评估环境**: Windows 11, Python 3.12  
**知识库**: kb_9 (38 个文档片段，网络配置技术文档)  
**LLM**: MiMo v2 Pro (OpenAI 兼容接口)  
**Embedding**: shibing624/text2vec-base-chinese (768 维)  
**检索方式**: 混合检索 (向量相似度 + BM25 关键词匹配)

---

## 一、评估概述

本次评估从三个维度对 IntraAI RAG 系统进行了全面测试：

| 维度 | 测试项 | 测试数据量 |
|------|--------|-----------|
| 检索质量 | Hit Rate, MRR, Recall, Precision, NDCG | 12 条查询 |
| 生成质量 | 忠实度, 关键词覆盖, 拒答能力 | 10 条查询 (7 可答 + 3 不可答) |
| 端到端 | 延迟, 完整性, 拒答, 多轮对话 | 3 条完整性 + 3 条拒答 + 2 组多轮对话 |

**测试数据集**: 基于知识库实际内容（PAP 认证网络配置文档）手工构建，包含问题、标准答案、相关文档片段索引。

### 2.3 关键发现

**优势**:
- **Hit Rate@5 = 91.67%**: 12 条查询中 11 条在 top-5 结果中命中了至少 1 个相关文档
- **MRR = 0.7778**: 第一个相关文档平均排名较高，用户通常能在前 2-3 个结果中找到答案
- **检索延迟低**: 平均 ~45ms（首次 719ms 因模型加载，后续稳定在 40-55ms）

**不足**:
- **Recall@5 = 35.93%**: 相关文档召回率偏低，主要因为部分查询的 ground truth 涉及多个分散片段（如 Q06 涉及 18 个片段），top-5 无法全部覆盖
- **Precision@5 = 30.0%**: 检索结果中约 30% 是相关文档，其余为语义相近但非直接相关的片段
- **Q09 完全失败**: "认证通过后 ping 测试结果" 未命中任何相关文档（chunk 32-37），因为问题措辞与文档中的具体 ping 输出文本差异较大

**根因分析**:
1. Recall 偏低是因为 ground truth 标注粒度细（精确到具体 chunk），而实际应用中只需检索到部分关键片段即可回答
2. Q09 失败是因为 "认证通过后 ping 测试结果" 与文档中 "5 packet(s) transmitted, 5 packet(s) received, 0.00% packet loss" 的表述差异大，向量相似度和 BM25 均未匹配
3. 混合检索（向量 + BM25）对精确数字/命令的匹配能力有限，建议增加精确匹配层

### 3.3 拒答能力分析

| 问题 | 是否拒答 | 回答摘要 |
|------|---------|---------|
| 公司食堂几点开门？ | ✓ 正确拒答 | "根据现有知识库，我无法回答这个问题。参考资料主要涉及网络技术配置..." |
| 如何申请年假？ | ✓ 正确拒答 | "根据现有知识库，我无法回答这个问题。" |

### 3.4 关键发现

**优势**:
- **拒答能力优秀 (100%)**: 所有 3 个不可回答问题均被正确识别并拒答，且拒答时给出了合理的解释（说明知识库内容范围）
- **忠实度中等 (0.42)**: 答案中的关键信息约 42% 可在上下文中找到依据，说明模型基本遵循了 "只根据参考资料回答" 的指令
- **答案结构化**: 模型倾向于使用编号列表、Markdown 格式组织答案，可读性好

**不足**:
- **关键词覆盖率低 (0.08)**: 答案与标准答案的关键词重叠度很低，但这主要是因为：
  - 模型倾向于用自己的话重新组织答案，而非直接引用原文
  - 标准答案的关键词提取方式（按句号分割）不够精确
  - 这实际上反映了模型的 **生成能力** 而非缺陷
- **忠实度有提升空间**: 约 58% 的答案关键信息无法在上下文中找到，可能是模型在回答时进行了推理和补充
- **Q04 存在事实错误**: 回答 "认证失败时出现 0% packet loss"，实际应为 100% packet loss，说明模型可能混淆了认证成功/失败的场景

## 四、端到端评估

### 4.1 延迟指标

| 指标 | 值 |
|------|-----|
| **平均总延迟** | 18,187ms (~18s) |
| **P50 延迟** | 16,502ms (~16.5s) |
| **P90 延迟** | 26,255ms (~26s) |
| **平均检索延迟** | 67ms |
| **平均生成延迟** | 18,120ms (~18s) |

**延迟构成**:
- 检索阶段仅占总延迟的 **0.37%** (67ms / 18,187ms)
- 生成阶段占总延迟的 **99.6%**，是性能瓶颈
- 首次查询因模型加载延迟较高 (719ms 检索)，后续稳定

### 4.2 答案完整性

| 问题 | 答案长度 | 检索片段数 | 完整性评价 |
|------|---------|-----------|-----------|
| PAP 认证项目完整实施步骤 | 858 字 | 5 | ✓ 涵盖背景、配置、验证 |
| ping 失败如何排查 | 456 字 | 5 | ✓ 提供了排查步骤 |
| PAP 认证工作原理 | 84 字 | 5 | ✓ 简洁准确 |

**平均答案长度**: 466 字，适中偏长，适合企业内部知识问答场景。

### 4.3 多轮对话连贯性

**对话 1: 渐进式深入** (项目概述 → 协议 → 命令 → 验证)

| 轮次 | 问题 | 关键词命中 | 得分 |
|------|------|-----------|------|
| Turn 1 | 这个项目是做什么的？ | 4/4 (PAP, 认证, 互联, 安全) | 100% |
| Turn 2 | R1和R2之间用什么协议？ | 2/2 (PPP, PAP) | 100% |
| Turn 3 | 认证配置的具体命令是什么？ | 2/2 (ppp authentication-mode pap, ppp pap local-user) | 100% |
| Turn 4 | 配置完成后怎么验证？ | 1/3 (ping, packet loss, dis ip int brief) | 33% |

**对话平均**: 83%

**对话 2: 指代消解** (测试 "R2呢" 类省略问句)

| 轮次 | 问题 | 关键词命中 | 得分 |
|------|------|-----------|------|
| Turn 1 | R1的IP地址是多少？ | 2/2 (192.168.10.254, 10.10.10.1) | 100% |
| Turn 2 | R2呢？ | 1/2 (192.168.20.254, 10.10.10.2) | 50% |
| Turn 3 | PC之间能通信吗？ | 2/3 (ping, 0%, packet loss) | 67% |

**对话平均**: 72%

### 4.4 关键发现

**优势**:
- **多轮对话基本连贯**: 平均得分 78%，模型能理解上下文并保持对话连贯性
- **指代消解有效**: "R2呢？" 能正确理解为 "R2 的 IP 地址是多少"，虽然只命中 1/2 关键词但回答方向正确
- **检索极快**: 平均 67ms，用户体验上几乎是瞬时

**不足**:
- **生成延迟过高**: 平均 18 秒，P90 达 26 秒，用户体验较差。主要瓶颈在 MiMo API 的响应速度
- **Turn 4 验证问题得分低**: 模型回答验证方法时遗漏了 `dis ip int brief` 命令，可能因为检索结果中该信息分散在多个片段
- **指代消解不完美**: "R2呢？" 只命中 50% 关键词，回答可能遗漏了部分 IP 信息

### 5.1 评分总览

| 维度 | 指标 | 得分 | 评级 |

| 检索质量 | Recall@5 | 35.93% | ⭐⭐ |
| 检索质量 | NDCG@5 | 44.97% | ⭐⭐⭐ |
| 生成质量 | 忠实度 | 41.85% | ⭐⭐⭐ |
| 生成质量 | 拒答正确率 | 100% | ⭐⭐⭐⭐⭐ |
| 端到端 | 检索延迟 | 67ms | ⭐⭐⭐⭐⭐ |
| 端到端 | 生成延迟 | 18.1s | ⭐⭐ |
| 端到端 | 多轮连贯性 | 77.78% | ⭐⭐⭐⭐ |

### 5.2 优势

1. **检索命中率高 (91.67%)**: 绝大多数查询能在 top-5 结果中找到相关文档
2. **拒答能力优秀 (100%)**: 能准确识别知识库外问题并给出合理解释
3. **检索速度极快 (67ms)**: 混合检索 + BM25 的实现效率很高
4. **多轮对话基本连贯 (78%)**: 能理解上下文和指代关系
5. **答案结构化程度高**: 使用 Markdown 格式，可读性好

### 5.3 待改进

1. **生成延迟过高 (18s)**: 建议：
   - 使用流式输出改善用户感知延迟
   - 考虑使用更快的模型变体（如 MiMo v2 Lite）
   - 优化 prompt 长度，减少 token 消耗

2. **召回率偏低 (35.93%)**: 建议：
   - 增加检索候选数（当前 50 → 100）
   - 优化分片策略，减少信息分散
   - 增加查询扩展（query expansion）层

3. **忠实度有提升空间 (41.85%)**: 建议：
   - 在 prompt 中更强调 "只使用上下文中的信息"
   - 减少 temperature（当前 0.7 → 0.3）
   - 考虑添加引用标记，让答案可溯源

4. **Q09 检索失败**: "认证通过后 ping 结果" 未命中相关文档，建议：
   - 增加同义词扩展（"ping 测试" ↔ "连通性测试"）
   - 对数字/命令类查询增加精确匹配层

5. **Q04 存在事实错误**: 模型混淆了认证成功/失败的 packet loss 比例，建议：
   - 在 prompt 中增加 "仔细区分正面和负面场景" 的指令
   - 提供更多上下文片段让模型对比

## 六、评估方法说明

### 6.1 测试数据集

- **检索评估**: 12 条查询，每条标注了相关文档片段索引（基于 kb_9 的 38 个 chunk）
- **生成评估**: 10 条查询（7 条可答 + 3 条不可答），标注了标准答案
- **多轮对话**: 2 组对话（渐进式深入 + 指代消解），共 7 轮
- **端到端**: 3 条完整性测试 + 3 条拒答测试

### 6.2 指标定义

| 指标 | 计算方式 |
|------|---------|
| Hit Rate@K | top-K 结果中是否至少包含 1 个相关文档 (0/1) |
| MRR | 第一个相关文档的排名倒数 (1/rank) |
| Recall@K | top-K 中相关文档数 / 总相关文档数 |
| Precision@K | top-K 中相关文档数 / K |
| NDCG@K | 归一化折损累积增益，考虑排序位置 |
| 忠实度 | 答案中关键信息在上下文中出现的比例 |
| 关键词覆盖率 | 答案对标准答案关键词的覆盖比例 |
| 拒答正确率 | 不可回答问题中正确拒答的比例 |

### 6.3 工具与框架

- **自定义评估脚本**: `backend/eval/run_eval_v2.py`
- **指标计算**: `backend/eval/metrics.py`
- **测试数据集**: `backend/eval/test_dataset.py`
- **Ragas 框架**: 已尝试集成但因 MiMo API 兼容性问题（不支持 `n` 参数、响应格式差异）未能成功运行，已使用自定义等效指标替代

### 6.4 局限性

1. 测试数据集规模较小 (12+10 条)，统计显著性有限
2. 知识库仅包含 1 个文档 (网络配置)，不代表多文档场景
3. 忠实度使用简易关键词匹配计算，非 LLM-as-judge
4. 未测试跨知识库查询、文档上传后检索等场景
5. 生成延迟受 MiMo API 服务端影响，非本地模型性能
